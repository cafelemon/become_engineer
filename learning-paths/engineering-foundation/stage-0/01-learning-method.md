# 01 学习方法

## 前置知识

无。

## 学习目标

学完本单元后，你应该能把一个模糊的学习想法，拆成可以执行、可以检查、可以复盘的小任务。

本单元不讲效率工具，也不讲学习鸡汤。我们只做一件事：让每次学习都留下证据。

## 学习顺序

1. 先理解基本学习循环。
2. 再练习把大目标改成小目标。
3. 然后用主动练习留下产出。
4. 最后记录问题并做阶段复盘。

## 你要养成的基本循环

每次学习都按这个循环走：

```text
目标 -> 操作 -> 结果 -> 问题 -> 下一步
```

- 目标：这次到底要学会什么。
- 操作：你实际做了哪些步骤。
- 结果：你看到了什么产出。
- 问题：哪里没懂，哪里失败了。
- 下一步：下次从哪里继续。

如果一段学习没有结果、没有问题、没有下一步，它很容易变成“我好像看过了”。

## 第一步：把目标写小

不要这样写：

```text
我要学编程。
```

这句话太大，今天无法验收。

改成这样：

```text
今天我要理解什么是目录、路径和扩展名，并能判断一个路径是相对路径还是绝对路径。
```

一个好目标通常满足三个条件：

- 今天能完成。
- 完成后能展示一个结果。
- 别人能判断你是否完成。

## 第二步：把学习变成主动练习

只看内容是不够的。每个小单元都至少要做一个动作：

- 写一段解释。
- 画一个简单结构。
- 判断几个例子。
- 修改一个文件。
- 记录一次错误。

例如学习“文件系统”时，不只记住“目录就是文件夹”，还要能写出一个项目目录结构，并说明每个文件放在哪里。

## 第三步：记录问题

问题不是失败，它是下一步的入口。

记录问题时不要只写：

```text
这里不懂。
```

要写得更具体：

```text
我不确定 `docs/README.md` 里的 `docs` 是目录名，还是文件名的一部分。
```

一个好问题通常包含：

- 你正在学哪一节。
- 哪个词、例子或步骤卡住了。
- 你已经尝试怎么理解。

## 第四步：做阶段复盘

每学完一个单元，花几分钟回答四个问题：

1. 我这节学会了什么。
2. 我留下了什么产出。
3. 我现在最容易犯什么错。
4. 下一节开始前我需要确认什么。

复盘不是写长文，而是防止自己带着混乱继续往下走。

## 示例：一份合格的学习记录

```markdown
# 学习记录：阶段 0 单元 1 学习方法

## 目标

理解“目标 -> 操作 -> 结果 -> 问题 -> 下一步”的学习循环。

## 操作

- 阅读单元 1。
- 把“我要学编程”改写成一个今天能验收的小目标。
- 按模板写了一份学习记录。

## 结果

我能写出一个更小的目标：今天理解目录、路径和扩展名。

## 问题

我还不确定后续每一节的练习要放在哪里记录。

## 下一步

进入单元 2 文件系统，练习写项目目录结构。
```

## 实践练习

### 练习 1：改写目标

把下面三个目标改写成“今天能完成、能展示结果、能判断是否完成”的版本。

```text
1. 我要学工程。
2. 我要学 Git。
3. 我要学 Python。
```

合格示例：

```text
我要学 Git。
-> 今天我要理解 `git status` 的作用，并能根据输出判断当前仓库有没有未保存到提交里的改动。
```

### 练习 2：写一份学习记录

用下面模板写一份你自己的学习记录。

```markdown
# 学习记录：阶段 0 单元 1

## 目标


## 操作


## 结果


## 问题


## 下一步


```

## 常见错误与排查

| 错误 | 为什么有问题 | 怎么改 |
| --- | --- | --- |
| 目标太大 | 今天无法判断是否完成 | 改成一个可以展示结果的小目标 |
| 只看不做 | 看完容易忘，也无法发现卡点 | 每节至少留下一个练习产出 |
| 问题太模糊 | 下次不知道从哪里继续 | 写清具体词、例子或步骤 |
| 复盘写成流水账 | 信息多但不可用 | 只回答学会什么、产出什么、错在哪里、下一步是什么 |

## 完成标准

完成本单元需要同时满足：

- 能解释为什么“我要学编程”不是一个好目标。
- 能把一个大目标改写成今天能验收的小目标。
- 写出一份包含目标、操作、结果、问题、下一步的学习记录。
