# AI Coding Agent 工作协议

## 总原则

这个项目优先追求最终学习效果，而不是拘泥于当前工具是否已经具备。需要新工具、依赖或插件时，AI Coding Agent 应说明原因并申请安装或授权。

AI 是默认协作工具，但 AI 输出始终是待验证输入。AI 可以提高脚手架、测试、文档、图表和排错效率，学习者与维护者仍然负责需求边界、关键判断、人工修改和最终验收。课程内容遵循[课程内容规范](08_content_standard.md)。

## 项目启动规则

新项目或新模块先写文档，再写实现。文档至少包含：

- 项目总览
- 产品需求
- 版本规划
- 技术架构
- 验收清单
- AI Coding 工作协议
- 进度记录
- 决策记录

## 内容边界

- 公开内容必须适合进入 GitHub。
- `个人学习/` 是本地私有目录，不得提交。
- 不把账号、密钥、课程登录信息、下载资料原件写入公开仓库。
- 不直接搬运受版权保护内容。

## 工作方式

- 先理解当前仓库结构，再修改文件。
- 优先保持文档、目录和命名的一致性。
- 所有课程、笔记、练习和项目先定位到[面向小白的统一学习路线](../learning-paths/beginner-roadmap.md)，不得绕过前置阶段。
- 六条主线用于组织内容，不得被解释为六条可任选的起步路线。
- 外部来源已登记不等于课程已完成；按当前阶段只取最小必要内容并独立重写。
- 对学习资源做筛选、归类、纠错和摘要，而不是简单堆叠。
- 遇到不确定知识点时标记为待验证。
- 优先使用官方资料核查版本、接口、安全和行为；无法只靠文档确认时，用最小实验或测试验证。
- 生成课程内容时先确定课程类型、能力目标和验收证据，再选择 Mermaid、表格、代码、输出或正反例。
- 新建或重构正式课程时，必须使用仓库内 `$become-engineer-course-authoring` Skill；它是任务步骤、知识卡、检索测试和站点接入的唯一生产流程源。
- 可复制代码必须写明文件、环境、依赖、运行命令、预期输出和失败路径，不用省略号隐藏关键逻辑。
- AI 生成的代码必须经过人工审阅、实际运行、关键逻辑解释、主动行为修改和验证后才能作为学习证据。
- 算法、数据结构、语言内存机制、模型关键步骤、权限和安全逻辑设置手动检查点。
- 微练习留在课程正文；连续课程形成阶段作品；长期项目跨课程演进，不恢复“一篇内容一个项目”。
- 对外发布前检查隐私、版权和准确性。

## AI 协作证据

课程或项目允许使用 AI 生成大部分重复代码，但验收至少保留：

- 任务目标、输入输出和约束。
- AI 产出摘要及人工接受、拒绝或修正的内容。
- 学习者主动修改的至少一个可观察行为。
- 实际运行的命令、测试、输出或指标。
- 一次基于错误信息、日志或失败测试的排查过程。
- 尚未验证的版本、兼容性、成本或安全风险。

不要求公开完整私人对话，也不能把“AI 说可以”作为验证结果。

## 推荐输出格式

资源条目应尽量包含：

- 名称
- 链接
- 类型
- 语言
- 适合阶段
- 前置知识
- 推荐理由
- 主要风险或缺点
- 学习建议
- 验证状态

学习笔记应尽量包含：

- 学习目标
- 前置知识和统一路线位置
- 可顺序执行的步骤
- 核心概念
- 常见误区
- 常见错误和排查方式
- 实践练习
- 客观验收标准
- 下一章节或并行分支
- 延伸资源
- 自测问题

课程正文、阶段作品和项目里程碑还应按[课程内容规范](08_content_standard.md)记录视觉设计、可复现代码、AI 协作证据、手动检查点和三级实践关系。
