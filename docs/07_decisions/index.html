    决策记录¶

    2026-06-25: 仓库采用公开学习平台定位¶

    决策¶

    本仓库定位为公开学习平台，覆盖工程开发、数据库、AI、LLM 和 Agent 等方向。

    原因¶

    项目既服务个人长期学习，也希望形成面向公众可阅读、可复用、可纠错的知识沉淀。

    影响¶

    • 文档需要面向外部读者，而不是只写给自己。
    • 资源收集需要保留来源和质量判断。
    • 笔记需要从个人草稿整理为公开表达。

    2026-06-25: 个人学习资料不进入公开仓库¶

    决策¶

    本地 个人学习/ 目录作为私有资料区，加入 .gitignore，不提交到 GitHub。

    原因¶

    个人学习资料可能包含未整理草稿、下载资料、隐私信息和版权风险内容，不适合公开。

    影响¶

    • 公开内容需要从私有目录中筛选、整理后再进入仓库。
    • 提交前必须检查 git status，确认私有目录未被跟踪。

    2026-06-25: 初期以 Markdown 仓库为主¶

    决策¶

    项目初期不急于搭建复杂站点，先使用 Markdown 和目录结构沉淀内容。

    原因¶

    当前最重要的是建立内容体系、筛选标准和长期维护节奏。

    影响¶

    • 后续可以平滑迁移到 VitePress、Docusaurus 或 MkDocs。
    • 现阶段优先完善内容结构和质量标准。

    2026-06-25: 采用四层六线结构¶

    决策¶

    项目采用四层六线结构组织公开内容。

    四层为学习地图、学习路线、项目实践、资源评测。六线为工程基础、编程语言、Web 全栈、CS 核心、AI 基础、LLM/Agent。

    原因¶

    单纯资源列表容易堆链接，单纯课程路线又难以承载个人学习沉淀。四层六线可以同时保留全局导航、阶段路线、实践验证和资源判断。

    影响¶

    • 新增内容必须先映射到主线和内容层级。
    • resources/ 记录资源事实，reviews/ 记录资源判断。
    • 从 个人学习/ 提炼的内容优先进入 publications/、notes/、projects/ 或 reviews/。

    后续修订¶

    2026-06-25 的“面向小白采用唯一顺序化路线”决策补充了本决策：六条主线继续用于分类，但不再作为并列可选的学习顺序。

    2026-06-25: 新增 content-inbox 作为素材收纳层¶

    决策¶

    新增 content-inbox/，用于登记未整理素材、所属主线、处理状态和下一步动作。

    原因¶

    当前素材数量多且来源混杂，直接进入正式目录会导致公开内容质量不稳定。Inbox 可以先承接混乱资料，再经过去重、去广告、纠错和项目筛选后进入正式目录。

    影响¶

    • Inbox 不保存访问密码、token、下载资料原件或受版权保护内容。
    • 素材必须先标注状态，再决定进入 resources/、reviews/、notes/、projects/ 或 publications/。
    • 项目推动是优先原则，但工程基础和纯理论内容不强制独立项目化。

    2026-06-25: 原始素材放入 source-materials 而非个人学习区¶

    决策¶

    新增 source-materials/ 作为项目级素材加工区。README.md 入库，raw/、working/、exports/ 作为本地 ignored 子目录。

    原因¶

    外部资料是项目内容生产来源，不属于个人学习区；但原文、广告、旧内容和中间产物也不适合直接公开。

    影响¶

    • 原始抓取内容放入 source-materials/raw/。
    • 清洗、去重、纠错过程文件放入 source-materials/working/。
    • 只有二次加工后的公开内容进入正式目录。

    2026-06-25: 内容加工以知识单元而非原文章为单位¶

    决策¶

    首个样品不逐篇清洗发布，而是将 4 篇 Function Calling 素材合并为一个新的 Tool Calling 知识单元。

    原因¶

    原素材存在大量主题重复、旧接口、广告和不安全示例。逐篇修改会保留原有结构缺陷，也难以形成清晰学习路径。

    影响¶

    • 加工区记录来源、重复关系、问题和取舍。
    • 公开内容采用独立结构和表达。
    • 一个知识单元可以产出笔记与聚焦练习，或服务于某个项目里程碑。

    2026-06-25: 首个 Tool Calling 练习采用离线优先¶

    决策¶

    安全销售查询练习默认使用离线模拟 Tool Call，OpenAI Responses API 作为可选适配器。

    原因¶

    学习者不应因为缺少 API Key、调用额度或特定模型权限而无法验证核心流程。离线模式也更适合自动测试参数和权限边界。

    影响¶

    • 核心项目只使用 Python 标准库。
    • 真实模型 SDK 和环境变量限制在项目目录。
    • 不把付费 API 调用作为验收条件。

    2026-06-25: 数据工具暴露业务能力而非任意 SQL¶

    决策¶

    样品只提供 get_monthly_sales，不提供 execute_sql。

    原因¶

    模型生成任意 SQL 后直接执行会扩大读取、写入、资源消耗和提示注入风险。固定业务工具更容易做权限、参数和结果控制。

    影响¶

    • SQL 结构由代码维护，模型参数只作为绑定值。
    • SQLite 使用只读连接。
    • 写操作和高风险工具未来默认需要更严格权限及人工确认。

    2026-06-25: 历史决策：单一大模型素材的适用边界¶

    决策¶

    《大模型全部学习笔记》只用于精做 Python、AI 基础和 LLM/Agent。工程基础、Web 全栈、CS 核心和 C++ 只登记缺口，不调整六条主线。

    原因¶

    素材在机器学习、深度学习和 Agent 上体量很大，但 Git、工程方法、Web、CS 和 C++ 内容零散，不能把标题命中当成完整课程。

    影响¶

    • 新素材先映射到六条主线和能力依赖图。
    • llama.cpp 不计入 C++ 基础。
    • FastAPI、Flask 和 Codex 案例不计入完整 Web 或工程课程。

    后续修订¶

    CS DIY和FastAPI等来源已完成目录登记，因此公开口径改为“来源已登记、尚未加工到该阶段”。单一素材的覆盖不足仍然成立，但不再表述为整条主线没有来源。

    2026-06-25: 项目跨笔记演进¶

    决策¶

    项目不与单篇笔记一一对应。一个项目由多个里程碑组成，每个里程碑关联前置课程、笔记、代码产出和验收结果。

    原因¶

    真实能力需要多个知识点共同完成。为每篇笔记创建项目会产生大量碎片，框架教程也会被误当成独立成果。

    影响¶

    • 建立五个项目主干，不为每个框架和章节建立项目。
    • 项目型笔记必须说明推进了哪个里程碑。
    • 纯理论笔记说明能力去向即可。
    • 聚焦实验进入 exercises/。

    2026-06-25: 基础课完整前置并在项目中持续回链¶

    决策¶

    工程、语言、数学和模型原理中必要的基础内容先形成完整课程；项目实施时继续链接回这些基础课，并按具体问题深化。

    原因¶

    完全项目驱动容易使知识碎片化，学完基础后完全脱离项目又会失去关联。两者需要同时存在。

    影响¶

    • 项目入口必须列出前置课程。
    • 项目笔记必须列出实际使用的基础知识。
    • 尚未加工完成的基础课程必须标明来源状态、加工进度和所在阶段，不得伪装为已完成课程。

    后续修订¶

    本决策中的“完整前置”已被“面向小白采用唯一顺序化路线”细化：语言前先完成必要工程基础，测试、CI、部署等能力在有代码和项目后持续深化。

    2026-06-25: 大型外部资料先建目录再按需取用¶

    决策¶

    CS 自学指南和 FastAPI 官方文档当前只做目录级梳理，不下载完整站点或仓库。具体路线章节或项目里程碑需要时，再拉取最小必要页面和示例。

    原因¶

    现有问题不是资源不足，而是资料分散且缺少统一路线。提前下载大量内容会增加重复、更新和版权管理成本。

    影响¶

    • content-inbox/external-source-catalog.md 记录来源结构、用途、许可和触发条件。
    • CS 自学指南主要用于课程筛选，不复制其完整路线。
    • FastAPI 主要作为官方事实来源和项目实现参考。
    • 每次取用记录 URL、日期、版本、许可和选取原因。
    • 外部原文如需本地处理，仍进入 ignored 的 source-materials/raw/。

    2026-06-25: 面向小白采用唯一顺序化路线¶

    决策¶

    全仓库采用唯一默认学习顺序：

    工程基础前置
  -> Python 起步
    -> Python/C++ 双主修
      -> CS 最小核心
        -> Web 与 CS 深化并行
          -> AI 基础
            -> LLM/Agent

    Java作为完整补充路线，JavaScript/TypeScript主要在Web阶段进入。

    原因¶

    六条并列主线适合维护者组织内容，却要求小白自己判断顺序。项目希望读者从第一步开始照着操作，也能逐阶段完成。

    影响¶

    • learning-paths/beginner-roadmap.md 成为唯一顺序基准。
    • 六条主线负责分类和阶段内容，不独立定义全局顺序。
    • 工程基础分为语言前必修和项目中持续深化。
    • Python先建立共同编程概念，再与C++并行主修。
    • CS最小核心完成后才启动Web，并在Web期间深化CS。
    • 项目按阶段解锁，专项地图和素材目录不能提前分配后期任务。

    2026-06-25: Docker、算法和数据库采用分层学习口径¶

    决策¶

    • Docker进入工程基础，但语言前只要求最小认知：镜像、容器、端口、挂载、环境变量和常用查看命令。
    • Dockerfile、docker compose、数据库容器、服务部署、监控和回滚在后续项目中逐步深化。
    • 算法拆成两层：CS核心学习通用算法与复杂度；AI基础学习机器学习、深度学习、NLP/Transformer和强化学习等模型算法。
    • 数据库实践优先使用 PostgreSQL 和 psql；MySQL作为常见生产数据库对照；SQLite只用于轻量本地练习或特定样例。

    原因¶

    Docker过早深入会拖慢小白进入编程，但后续部署和复现离不开容器认知。算法如果只写一个词，容易混淆刷题、数据结构和模型训练。数据库如果只写SQL，也会忽略真实工程中连接、迁移、事务、索引和部署工具链的差异。

    影响¶

    • 工程基础课程需要加入Docker最小认知章节，但不把容器部署设为语言前门槛。
    • CS核心课程必须覆盖通用算法，不把机器学习内容提前塞进CS阶段。
    • AI基础课程必须明确模型算法的前置和边界，NLP/Transformer作为LLM/Agent关键桥梁。
    • Web与CS数据库内容默认围绕PostgreSQL/psql组织，再补MySQL差异。

    2026-06-25: 小白课程单元采用可执行文档契约¶

    决策¶

    每个正式课程单元必须包含前置知识、明确顺序、命令或示例、练习或检查、常见错误与排查、客观验收和下一步。

    原因¶

    只有概念总结和资源链接仍然要求学习者自行组织学习，不能满足“挨着做也能行”的目标。

    影响¶

    • 课程模板增加解锁条件、错误排查和阶段导航。
    • 项目章节持续回链工程、语言和CS基础。
    • 后续课程正文需要实际运行验证，不能只做目录规划。

    2026-06-25: 公开阅读体验采用 MkDocs Material¶

    决策¶

    本仓库采用 MkDocs Material 生成公开网站。Markdown 继续作为唯一内容源，mkdocs.yml 直接以仓库根目录作为文档源，并通过排除规则保护私有目录和原素材加工区。

    原因¶

    内容会持续增长，单靠 GitHub 文件列表不适合新学习者按阶段阅读。MkDocs Material 可以提供导航、目录、搜索、深浅色主题和 GitHub Pages 发布能力，同时不改变现有 Markdown 写作方式。

    影响¶

    • 新增 mkdocs.yml、requirements.txt 和 GitHub Actions 发布 workflow。
    • 网站发布目标为 https://cafelemon.github.io/become_engineer/。
    • 个人学习/、source-materials/raw/、source-materials/working/、source-materials/exports/ 不进入站点构建。
    • 后续新增正式课程时需要同步更新 mkdocs.yml 导航。
