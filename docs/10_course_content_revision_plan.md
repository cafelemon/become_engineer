# 课程内容架构修订总计划

## 文档状态

- 状态：样板、V2 规范和生产链均已固化；原有 55 节与 Web 起步四课已经完成，Web 核心第一课进入正式课程。
- 适用范围：课程正文、课程视觉、可复现示例、阶段作品、长期项目和课程内小码助教。
- 当前正式基线：[课程内容规范 V2](08_content_standard.md)与升级后的课程生产 Skill；所有正式课程使用 V2 语义协议，运行时继续保留 V1 兼容能力。
- 本计划是后续修订的唯一总路线；样板仍不进入正式站点，新课只有在正文、项目、助教、登记和导航同时就绪后才计入正式课程。

## 为什么修订

现有 55 节课程已经完成任务路线、可观察结果、主动修改、失败路径、迁移验收和知识库助教覆盖，解决了“只有说明、没有练习”的问题。但把 5–7 步任务作为所有课程的统一开场，也产生了新的学习障碍：

- 小白尚未形成概念画面，就需要连续执行多个步骤，容易只复制命令而不理解。
- 工具、编程、CS、算法、系统、AI 和项目课程使用近似节奏，无法体现不同知识的最佳学习方式。
- Mermaid 和代码数量增加了，但真实界面、状态变化、执行过程和成果预览仍然不足。
- 阶段作品和长期项目已经存在，却没有在每节课中持续显示“当前版本、本课增量和下一版本”，项目叙事不够可见。
- 新手帮助、深入原理和求职训练混在同一阅读流中，造成部分内容过浅、部分内容又突然过深。

因此，后续目标不再是统一“任务先行”，而是按课型组织“理解驱动、小步实践、项目贯穿、分层深化”。任务仍然保留，但只在学习者获得必要方向感和最小心智模型后出现。

## 目标学习体验

课程按实际课型组合以下学习环节，不固定步骤数量：

1. **兴趣与方向**：用真实问题、最终效果或产品预览说明为什么值得学。
2. **最小心智模型**：用一张功能型视觉、一个类比或一组对照建立概念画面。
3. **微型例子**：提供可完整看懂的小例子，先预测结果，再运行验证。
4. **引导复现**：说明文件、环境、操作或命令、预期输出和首次失败恢复。
5. **主动修改**：学习者必须改变输入、行为、边界或结构，不能只复制教师代码。
6. **解释与排错**：在操作需要的位置解释机制，并通过受控错误学习定位问题。
7. **项目里程碑**：说明本课给阶段作品或长期项目增加了什么能力和证据。
8. **深化与迁移**：按需提供原理、性能、工程取舍、求职追问和新场景迁移。
9. **总结与下一步**：要求学习者用自己的话解释，并指向项目下一版本或后续课程。

四类用户共享同一主叙事，通过三类可识别、可折叠但可直接访问的内容调整深度：

- **新手补给**：术语、安装、界面位置、第一次操作和常见误区。
- **深入理解**：实现机制、边界、性能、复杂度和替代方案。
- **求职训练**：机考迁移、故障分析、项目追问、设计取舍和证据表达。

## 八类课程草案

| 课型 | 合适开场 | 核心教学组织 | 必要视觉与证据 |
| --- | --- | --- | --- |
| 工具操作 | 展示软件界面和本课完成结果 | 找到入口、理解用途、完成最小操作、识别成功状态 | 带编号截图、界面前后对照、实际文件或命令结果 |
| 编程起步 | 展示几行代码产生的可见结果 | 心智模型、微型代码、预测、运行、修改、错误恢复 | 变量或控制流图、代码与输出、项目版本增量 |
| CS 概念 | 从程序为何这样运行的问题开始 | 具体对象、表示方法、状态变化、抽象规则、边界 | 内存／结构图、状态表、双实现对照 |
| 算法与数据结构 | 先人工解决一个具体问题 | 问题建模、状态演进、规律、伪代码、实现和复杂度 | 逐步轨迹、可回放状态、边界与复杂度证据 |
| 系统与设备 | 从故障、资源、时序或性能现象开始 | 系统边界、最小实验、机制、调试与可靠性 | 生命周期、调用链、接口时序、日志和真机／仿真证据 |
| AI 实验 | 从数据、目标和基线结果开始 | 数据划分、基线、训练、验证、误差分析和复现 | 数据流、指标图、实验配置和结果对照 |
| Web／LLM／Agent 应用 | 从一次请求、失败响应或受限工作流开始 | 输入输出契约、调用链、错误边界、评估和恢复 | 系统序列、请求响应、状态与可观测证据 |
| 项目整合 | 展示当前版本与新增需求 | 设计选择、增量实现、测试、演示、复盘和下一版本 | 产品预览、版本差异、测试报告和求职叙事 |

课型只决定教学组织，不改变课程在完整课程地图中的层级、角色和前置关系。

## 功能型视觉与交互原则

- 工具课使用真实界面截图，并标记入口、操作区域和成功状态；截图注明平台或版本，避免把过期界面当作永久事实。
- 概念课使用原创关系图、执行图、生命周期图和对照表，每张图只回答一个明确问题。
- 算法课在状态变化难以靠静态图解释时使用前进、后退、重置的逐步播放器，同时提供静态轨迹表降级。
- 系统、AI、LLM 和 Agent 课程优先展示数据流、状态边界、失败路径和评估证据，不用装饰图代替机制。
- 项目课展示运行效果、版本前后差异、测试结果和可保存证据。
- 所有视觉必须有标题、正文解释和可理解的文本替代；颜色不是唯一信息载体。
- 交互支持键盘、390px 窄屏、深浅色和减少动画；JavaScript 失效时完整知识、代码和静态证据仍可学习。
- 不设置“每课必须几张图”的形式指标，以是否显著降低理解成本作为取舍标准。

## 可复现代码与运行边界

- 示例代码保存为独立源文件，通过现有 `pymdownx.snippets` 引入正文，避免课程代码与可执行代码漂移。
- 每个可运行示例说明运行时版本、入口文件、依赖、命令、输入、预期输出、失败恢复和验证方式。
- Python 起步和基础算法的微型示例可以按需提供浏览器运行器；正式练习和项目仍需本地运行并保存证据。
- 浏览器运行器只在用户点击后加载，放入 Web Worker，限制为课程允许的微型程序；执行超时后终止并重建 Worker。
- 浏览器示例不自动安装第三方包，不替代环境、文件、终端、测试和工程工具训练。
- C、C++、系统、Web 服务和设备课程使用真实本地工具链；设备课程继续遵守“仿真入门、真机毕业”。

## 项目贯穿体系

### 单课到阶段作品

每节进入连续成果线的课程都显示：

- 所属项目及当前版本。
- 上一版本已经具备的能力。
- 本课新增的可观察能力。
- 学习者必须保存的运行、测试、排错或解释证据。
- 下一节将演进的版本。

每 3–6 节相关课程形成一个可独立运行、可解释、可验收的阶段作品。单模块成果继续放入 `exercises/`；推进长期项目时只更新对应项目里程碑，不重复创建目录。

### 分方向长期项目

| 路线 | 长期项目方向 | 主要叙事 |
| --- | --- | --- |
| 共同基座 | 学习进度报告器 | 从微型 Python 程序演进到可测试、可配置的阶段作品 |
| 应用与 Web | 工程学习工作台 | 从命令行报告进入 API、数据库和前端界面 |
| C++ 与系统 | 跨语言可追踪数据引擎 | 对照类型、内存、构建、性能、测试和可观测性 |
| AI 模型 | 可复现实验与模型评估系统 | 建立数据、训练、验证、误差分析和模型演进闭环 |

算法课程使用可追踪阶段实验，并将算法能力迁移到方向项目；不为统一外观强行建立一个覆盖所有课程的超级项目。

## 小码助教正式架构

正式运行时同时支持 V1 的 `lesson_id + step_id` 和 V2 的 `lesson_id + context_id`。V2 上下文类型包括：

```text
overview / concept / example / reproduce / modify
troubleshoot / project / deepen / career
```

正式行为：

- 根据当前阅读环节推荐概念解释、运行帮助、排错、项目或深化问题。
- 欢迎语从“先自己试一次”调整为“先理解，再一起动手”。
- 知识卡继续提供诊断、两层提示、局部示例、主动展开答案和来源。
- 未命中仍明确降级，不接模型、不猜测、不保存原始提问。
- 知识库只能选择 `steps` 或 `contexts` 其中一种；运行时自动识别，防止未迁移课程失效。
- V1/V2 使用独立存储键，只保存面板状态和提示层级；统一校验同时回归正式课与冻结样板。

## 样板验收基线

每节样板至少满足：

- 第一屏说明真实问题、结果预览和项目关系，不以多步骤任务列表开场。
- 学习者能用自己的话说明核心概念和本课用途。
- 微型例子可以实际运行，代码、命令和预期输出一致。
- 包含一次主动修改、一次受控失败或边界观察，以及明确恢复方法。
- 展示项目上一版本、本课增量、保存证据和下一版本。
- 新手、深入和求职内容层次清楚，不复制三份正文。
- 功能型视觉显著帮助理解，并具备文本和静态降级。
- 小码问题与当前学习环节匹配，未知问题诚实降级。
- 桌面、390px 手机、深浅色、键盘、减少动画和禁用 JavaScript 可用。

批次 A 样板完成后邀请四类目标用户各至少 3 人验证：至少 10 人能在两分钟内理解课程方向和第一步，至少 10 人复现最小例子，至少 9 人不展开完整答案完成主动修改，至少 9 人能说明本课给项目增加的能力和证据。
